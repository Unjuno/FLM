# FLM Docs Plan Review - 問題点抽出レポート
> Status: Review | Audience: Architecture & Engineering leads | Updated: 2025-11-18

## 背景

本レポートは、前回のプロトタイプ（`archive/prototype/`）の失敗を踏まえ、現在のdocs計画（`docs/PLAN.md` および関連仕様書）を深くレビューし、問題になる可能性のある点を抽出したものです。

### 前回の失敗要因（archive/prototype/ から抽出）

1. **機能の肥大化**: 機能が肥大化し保守が困難になった
2. **アーキテクチャの問題**: Node/Express実装で、Rustコアとの統合が不十分
3. **責務の逆転**: CLI/UI/Proxy間で責務が逆転していた可能性
4. **重複実装**: 同じロジックが複数箇所に実装されていた
5. **セキュリティ問題**: APIキーの暗号化強度、タイミング攻撃脆弱性、ログに機密情報が含まれる可能性

---

## 🔴 重大な問題（即座に対応が必要）

### 1. セキュリティ設計の不備

#### 1.1 APIキーのハッシュ化方式が未定義
**問題点**:
- `CORE_API.md` では「ハッシュ＋ソルトで保存」と記載されているが、具体的なアルゴリズム（Argon2id/BCrypt/PBKDF2）が未指定
- 前回の失敗（SEC-001, SEC-002）で指摘されたタイミング攻撃対策が明示されていない
- `SecurityService::create_api_key` の実装詳細が不明確

**影響範囲**: 認証プロキシ全体、APIキー管理機能

**推奨対応**:
- `docs/CORE_API.md` に APIキーハッシュ化の詳細仕様を追加（Argon2id推奨、パラメータ明記）
- タイミング攻撃対策として定数時間比較（constant-time comparison）を必須化
- `SecurityService` の実装例または参照実装を明記

#### 1.2 ログに機密情報が含まれるリスク
**問題点**:
- `PROXY_SPEC.md` では「request_id, timestamp, client_ip, api_key_id, endpoint, engine_id, latency_ms, status, error_type」をログに記録とあるが、リクエストボディやエラーメッセージの扱いが不明確
- 前回の失敗（SEC-006）で指摘された「ログにAPIキーやトークンが含まれないか」の対策が未記載
- エラーログにスタックトレースが含まれる可能性が考慮されていない

**影響範囲**: 監査ログ、セキュリティ監査

**推奨対応**:
- `docs/PROXY_SPEC.md` にログサニタイゼーション仕様を追加（機密情報のマスキングルール）
- エラーログの出力レベルとスタックトレースの扱いを明記
- ログファイルのアクセス権限設定（600相当）を `DB_SCHEMA.md` に追記

#### 1.3 セキュリティポリシーのバリデーション不足
**問題点**:
- `SecurityPolicy.raw_json` は任意のJSONを受け入れる設計だが、バリデーション仕様が不明確
- Phase1/2では `"default"` のみ扱うが、不正なJSONが設定された場合のエラーハンドリングが未定義
- IPホワイトリストのCIDR形式バリデーション、CORS設定の妥当性チェックが未記載

**影響範囲**: セキュリティポリシー適用、プロキシの動作

**推奨対応**:
- `docs/CORE_API.md` に `SecurityPolicy` のJSONスキーマバリデーション仕様を追加
- IPアドレス/CIDR形式のバリデーションルールを明記
- 不正なポリシー設定時のエラーコードとメッセージを定義

---

### 2. 責務境界の不明確さ

#### 2.1 Firewall Automation の責務が曖昧
**問題点**:
- `UI_MINIMAL.md` では「Tauri バックエンド（Adapter層）が OS 判定・昇格を実装」とあるが、Core APIとの境界が不明確
- `system_firewall_preview/apply` は Core API に含まれないとあるが、Proxy/Policy情報の取得は Core 経由という設計が複雑
- 前回の失敗（責務の逆転）を再現する可能性がある

**影響範囲**: UI実装、アーキテクチャの一貫性

**推奨対応**:
- `docs/CORE_API.md` に Firewall Automation の責務境界を明確化
- Core が提供する情報（Proxy status, Policy）と Adapter が実装する処理（OS判定、昇格実行）を明確に分離
- 依存関係図（`diagram.md`）に Firewall Automation の位置を追加

#### 2.2 ProxyService と ProxyController の責務重複
**問題点**:
- `CORE_API.md` では `ProxyService` が `ProxyController` trait を使用するとあるが、両者の責務が重複している可能性
- `ProxyService::start` が Axum サーバ起動を統括するとあるが、`ProxyController` の役割が不明確
- 前回の失敗（重複実装）を再現する可能性がある

**影響範囲**: Proxy実装、コードの保守性

**推奨対応**:
- `docs/CORE_API.md` に `ProxyService` と `ProxyController` の責務を明確に分離
- `ProxyService`: ビジネスロジック（設定管理、プロファイル管理）
- `ProxyController`: インフラ操作（プロセス起動/停止、ポート管理）
- 実装例または参照実装を追加

---

### 3. エラーハンドリングの一貫性不足

#### 3.1 エラーコード体系が未定義
**問題点**:
- `CLI_SPEC.md` ではエラーコード例（`ENGINE_NOT_FOUND`, `PROXY_ALREADY_RUNNING` 等）が記載されているが、体系的な定義がない
- 前回の失敗（ARCH-001, ARCH-005）で指摘された「エラーハンドリングの一貫性」が未解決
- Rust Core / CLI / Proxy / UI 間でエラー形式が統一されていない可能性

**影響範囲**: 全体のエラーハンドリング、デバッグ効率

**推奨対応**:
- `docs/CORE_API.md` にエラーコード体系を追加（カテゴリ別分類、階層構造）
- エラーレスポンス形式を統一（`{ code: String, message: String, details: Option<Value> }`）
- 各サービス（EngineService, ProxyService等）のエラー型を定義

#### 3.2 エラー遷移の未定義
**問題点**:
- `CORE_API.md` では `EngineStatus` の遷移規則は定義されているが、エラー発生時の復旧手順が不明確
- `ErrorNetwork` / `ErrorApi` から `RunningHealthy` への復帰条件が「連続成功が閾値を満たした際」とあるが、具体的な閾値が未定義
- プロキシ起動失敗時のロールバック手順が未記載

**影響範囲**: エラー回復、システムの堅牢性

**推奨対応**:
- `docs/CORE_API.md` にエラー復旧の詳細仕様を追加（リトライ回数、バックオフ戦略）
- `docs/ENGINE_DETECT.md` にエラー検出と復旧のフロー図を追加
- プロキシ起動失敗時のクリーンアップ手順を `PROXY_SPEC.md` に追加

---

## 🟡 中程度の問題（設計段階で解決推奨）

### 4. データベース設計の拡張性不足

#### 4.1 マイグレーション戦略の不備
**問題点**:
- `DB_SCHEMA.md` では `sqlx::migrate!()` を使用するとあるが、マイグレーションのロールバック戦略が未定義
- Phase1/2では `security_policies` に `id = "default"` のみ扱うが、将来の複数ポリシー対応時のマイグレーション手順が不明確
- 前回の失敗（ARCH-003）で指摘された「スキーマバージョン管理」が未解決

**影響範囲**: データベースの拡張性、バージョンアップ時の互換性

**推奨対応**:
- `docs/DB_SCHEMA.md` にマイグレーションロールバック戦略を追加
- スキーマバージョン管理の仕組みを明記（`schema_version` テーブルまたは `sqlx` の標準機能）
- 将来の複数ポリシー対応時のマイグレーション例を追加

#### 4.2 トランザクション処理の未定義
**問題点**:
- `DB_SCHEMA.md` ではトランザクション処理が明示されていない
- APIキーのローテーション（`rotate_api_key`）は複数テーブル更新が必要だが、トランザクション保証が不明確
- 前回の失敗（DB-002）で指摘された「トランザクション使用」が未解決

**影響範囲**: データ整合性、同時アクセス時の安全性

**推奨対応**:
- `docs/DB_SCHEMA.md` にトランザクション処理の仕様を追加
- `SecurityService::rotate_api_key` のトランザクション範囲を明記
- デッドロック回避のためのロック順序を定義

---

### 5. IPC/API設計の型安全性不足

#### 5.1 TypeScript-Rust間の型整合性が未保証
**問題点**:
- `UI_MINIMAL.md` では IPC 経由で Core API を呼び出すとあるが、型定義の共有方法が不明確
- 前回の失敗（ARCH-002）で指摘された「TypeScriptとRust間の型定義の整合性確認」が未解決
- Tauri IPC の型安全性を保証する仕組みが未記載

**影響範囲**: UI実装、型安全性、ランタイムエラー

**推奨対応**:
- `docs/UI_MINIMAL.md` に型定義の共有方法を追加（Tauri IPC型生成、JSON Schema からの型生成等）
- Rust側の型定義からTypeScript型を自動生成する手順を明記
- ランタイム型チェック（Zod等）の使用方針を追加

#### 5.2 APIキー平文の扱いが不明確
**問題点**:
- `CORE_API.md` では「作成直後のみ平文キーを表示」とあるが、UIへの受け渡し方法が不明確
- IPC経由で平文キーを送信する際のセキュリティ対策が未記載
- メモリ上での平文キーの扱い（ゼロクリア等）が未定義

**影響範囲**: セキュリティ、UI実装

**推奨対応**:
- `docs/CORE_API.md` に APIキー平文の扱い仕様を追加（IPC経由の暗号化、メモリクリア）
- UI側での平文キー表示後の破棄方法を明記
- セキュアなIPC通信の実装方針を追加

---

### 6. テスト可能性の不足

#### 6.1 モック化戦略が未定義
**問題点**:
- `FEATURE_SPEC.md` では「モック: Ollama API / OpenAI互換API を HTTP server mock で再現」とあるが、具体的なモック化方法が不明確
- Core API の trait（`LlmEngine`, `HttpClient`等）のモック実装方法が未記載
- 前回の失敗（QA-001, QA-002）で指摘された「テストカバレッジ」の改善策が不明確

**影響範囲**: テスト実装、CI/CD

**推奨対応**:
- `docs/CORE_API.md` に trait のモック実装例を追加
- `docs/CLI_SPEC.md` に統合テストのモック化手順を追加
- テスト用のHTTPサーバモックの実装例を追加

#### 6.2 エッジケースのテスト要件が不明確
**問題点**:
- Phase1の合格基準では「エンジン検出成功率100%」とあるが、エッジケース（ネットワーク断、タイムアウト、不正レスポンス等）のテスト要件が不明確
- プロキシ起動失敗時のテストシナリオが未定義
- 前回の失敗（QA-002）で指摘された「エラーケースのテスト」が未解決

**影響範囲**: テスト品質、システムの堅牢性

**推奨対応**:
- `docs/PLAN.md` のメトリクスセクションにエッジケーステスト要件を追加
- `docs/ENGINE_DETECT.md` にエラーケースのテストシナリオを追加
- プロキシ起動失敗時のテスト手順を `PROXY_SPEC.md` に追加

## 🟢 軽微な問題（実装時に注意）

### 7. ドキュメントの不完全性

#### 7.1 未確定事項の多さ
**問題点**:
- `CORE_API.md` の「7. 未確定事項」に ProxyConfig の ACME 設定が残っている
- `PROXY_SPEC.md` の「10. 未決事項」に ProxyService でのホットリロードが残っている
- Phase1開始前に未確定事項を減らすべき

**影響範囲**: 実装の遅延、仕様変更のリスク

**推奨対応**:
- Phase0で未確定事項をすべて確定させる
- 確定できない場合は「Phase3以降に延期」と明記し、Phase1/2の範囲から除外

#### 7.2 用語の統一不足
**問題点**:
- 「プロキシ」「Proxy」「proxy」の表記が統一されていない
- 「エンジン」「Engine」「engine」の表記が統一されていない
- 前回の失敗（DOC-002）で指摘された「ドキュメントの一貫性」が未解決

**影響範囲**: ドキュメントの可読性、開発効率

**推奨対応**:
- `docs/README.md` に用語集を追加
- 各ドキュメントで用語を統一（英語表記を優先、日本語は補足）

## 🎯 推奨される対応順序

### Phase 0（即座に対応）
1. **セキュリティ設計の確定**（1.1, 1.2, 1.3）
   - APIキーハッシュ化仕様の確定
   - ログサニタイゼーション仕様の追加
   - セキュリティポリシーのバリデーション仕様の追加

2. **責務境界の明確化**（2.1, 2.2）
   - Firewall Automation の責務境界の明確化
   - ProxyService と ProxyController の責務分離

3. **エラーハンドリングの統一**（3.1, 3.2）
   - エラーコード体系の定義
   - エラー遷移と復旧手順の定義

### Phase 0.5（Phase1開始前）
4. **データベース設計の拡張**（4.1, 4.2）
   - マイグレーション戦略の確定
   - トランザクション処理の仕様追加

5. **型安全性の確保**（5.1, 5.2）
   - TypeScript-Rust間の型定義共有方法の確定
   - APIキー平文の扱い仕様の追加

### Phase 1（実装と並行）
6. **テスト戦略の確立**（6.1, 6.2）
   - モック化戦略の実装
   - エッジケーステスト要件の追加

7. **ドキュメントの整備**（7.1, 7.2）
   - 未確定事項の確定または延期
   - 用語の統一

## 📝 結論

現在のdocs計画は、前回の失敗を踏まえた良い設計になっていますが、以下の点で改善が必要です：

1. **セキュリティ設計の詳細化**: APIキー管理、ログサニタイゼーション、セキュリティポリシーのバリデーションが不明確
2. **責務境界の明確化**: Firewall Automation と ProxyService/ProxyController の責務が曖昧
3. **エラーハンドリングの統一**: エラーコード体系とエラー遷移が未定義
4. **データベース設計の拡張**: マイグレーション戦略とトランザクション処理が不明確
5. **型安全性の確保**: TypeScript-Rust間の型定義共有方法が未定義

これらの問題を Phase 0 で解決することで、前回の失敗を回避し、堅牢な実装が可能になります。
