# FLM Docs Plan Review - 問題点抽出レポート
> Status: Review | Audience: Architecture leads | Updated: 2025-01-XX

## 背景

本ドキュメントは、`docs/PLAN.md` を中心とした再構築計画について、前回の失敗要因（機能肥大化・保守困難・責務逆転）を踏まえた設計検証を行い、破綻の可能性や実装上の問題点を抽出したものです。

## 前回の失敗要因（参考）

- 機能が肥大化し保守が困難になった
- CLI を子プロセスとして UI が呼び出す構造による責務逆転
- 重複実装の発生

## 検出された問題点

### 1. データベースマイグレーションの競合問題 【重要度: 高】

**問題:**
- `DB_SCHEMA.md` では、CLI/UI/Proxy の各プロセスが起動時に `sqlx::migrate!()` を実行すると記載されている
- 複数のプロセスが同時に同じ DB に対して migration を実行しようとすると、SQLite のロック競合が発生する可能性がある

**影響範囲:**
- CLI と UI を同時に起動した場合
- Proxy が起動中に CLI で設定変更を行った場合
- 複数の Proxy インスタンスを起動しようとした場合（想定外だが）

**具体的な問題箇所:**
- `DB_SCHEMA.md` 3章: マイグレーションの実行タイミング
- `CORE_API.md` 395行目: `*_Service::new()` での migration 実行

**推奨される対策:**
- SQLite の WAL (Write-Ahead Logging) モードを明示的に有効化
- Migration 実行時のロック機構（ファイルロックまたは DB ロック）を明記
- または、Migration を単一の初期化プロセスに集約し、他のプロセスは既に migration 済みの DB を前提とする設計に変更

**関連ドキュメント:**
- `docs/DB_SCHEMA.md`
- `docs/CORE_API.md`

---

### 2. エンジン検出キャッシュの整合性問題 【重要度: 中】

**問題:**
- `ENGINE_DETECT.md` では、キャッシュ TTL 30秒と記載されているが、複数のプロセス（CLI/UI/Proxy）が同時にキャッシュを更新する場合の整合性が不明確
- `config.db` の `engines_cache` テーブルを複数プロセスが同時に更新する場合の競合対策が記載されていない

**影響範囲:**
- CLI と UI を同時に使用した場合
- Proxy 起動時のエンジン検出と CLI での検出が同時に実行された場合

**具体的な問題箇所:**
- `ENGINE_DETECT.md` 3章: 共有ロジック（キャッシュ TTL 30秒）

**推奨される対策:**
- キャッシュ更新時のロック機構（DB ロックまたはファイルロック）を明記
- または、キャッシュ更新を単一のプロセス（例: CLI）に集約し、他のプロセスは読み取り専用とする設計に変更
- キャッシュの整合性チェック（`cached_at` の検証）を明記

**関連ドキュメント:**
- `docs/ENGINE_DETECT.md`
- `docs/DB_SCHEMA.md`

---

### 3. Firewall Automation の実装場所の曖昧さ 【重要度: 中】

**問題:**
- `UI_MINIMAL.md` では、Firewall Automation を `ipc.system_firewall_preview/apply` として IPC 経由で呼び出すと記載
- 一方で、「Core API に含まれない」「Tauri 側に実装する」と記載されており、実装場所が曖昧
- Core API と Adapter 層の境界が不明確

**影響範囲:**
- UI 実装時の混乱
- テスト戦略の不明確さ（Core API のテストに含めるべきか）

**具体的な問題箇所:**
- `UI_MINIMAL.md` 4章: IPC / HTTP フロー（Firewall Preview/Apply）
- `UI_MINIMAL.md` 6章: Firewall Automation IPC

**推奨される対策:**
- Firewall Automation を明確に Adapter 層（Tauri バックエンド）の責務として定義
- Core API との接点（Proxy/Policy 情報の取得）を明示的にリスト化
- IPC の実装詳細（Tauri コマンド定義、エラーハンドリング）を `UI_MINIMAL.md` または別ドキュメントに追記

**関連ドキュメント:**
- `docs/UI_MINIMAL.md`
- `docs/SECURITY_FIREWALL_GUIDE.md`
- `docs/CORE_API.md`

---

### 4. プロキシの状態管理とプロセス管理の不明確さ 【重要度: 高】

**問題:**
- `ProxyService` が複数のプロキシハンドルを管理すると記載されているが、同一ポートでの重複起動防止の仕組みが不明確
- Proxy が別プロセスとして起動する場合のプロセス管理方法（PID 管理、Graceful shutdown、プロセス監視）が詳細に定義されていない

**影響範囲:**
- 同一ポートでの重複起動によるエラー
- Proxy プロセスの異常終了時の復旧方法
- 複数の Proxy インスタンスを管理する場合の整合性

**具体的な問題箇所:**
- `CORE_API.md` ProxyService の定義
- `PROXY_SPEC.md` Proxy 起動/停止の仕様
- `CLI_SPEC.md` `flm proxy start/stop` の仕様

**推奨される対策:**
- ポート重複チェックの仕組みを明記（`ProxyService::start` での事前チェック）
- プロセス管理の詳細（PID ファイル管理、プロセス監視、異常終了時の復旧）を `PROXY_SPEC.md` に追記
- `ProxyHandle` の `pid` フィールドの使用方法を明記

**関連ドキュメント:**
- `docs/CORE_API.md`
- `docs/PROXY_SPEC.md`
- `docs/CLI_SPEC.md`

**推奨される対策:**

**関連ドキュメント:**
- `docs/UI_MINIMAL.md`
- `docs/CORE_API.md`

**問題:**

**推奨される対策:**
- ACME 証明書の自動更新プロセス（更新タイミング、失敗時の処理）を `PROXY_SPEC.md` に追記
- 自己署名証明書の生成・管理方法を `PROXY_SPEC.md` に追記
- `certificates` テーブルの使用方法を `DB_SCHEMA.md` に追記

**関連ドキュメント:**
- `docs/PROXY_SPEC.md`

### 9. エラーハンドリングとログ管理の統一性不足 【重要度: 中】

**問題:**
- 各コンポーネント（CLI/UI/Proxy）でのエラーハンドリングポリシーが統一されていない
- ログ管理の標準化（ログレベル、ログ形式、ログ保存場所）が詳細に定義されていない

**影響範囲:**
- デバッグの困難さ
- エラーメッセージの一貫性の欠如

**具体的な問題箇所:**
- `CLI_SPEC.md` エラー仕様（CLI のみ）
- `PROXY_SPEC.md` エラー・ログポリシー（Proxy のみ）
- UI のエラーハンドリングが `UI_MINIMAL.md` に記載されていない

**推奨される対策:**
- エラーハンドリングポリシーを `CORE_API.md` または別ドキュメントに統一して定義
- ログ管理の標準化（ログレベル、ログ形式、ログ保存場所）を `PLAN.md` または別ドキュメントに追記

**関連ドキュメント:**
- `docs/CLI_SPEC.md`
- `docs/PROXY_SPEC.md`
- `docs/UI_MINIMAL.md`
- `docs/CORE_API.md`

## 優先度別まとめ

### 【高優先度】実装前に解決必須
1. データベースマイグレーションの競合問題
2. プロキシの状態管理とプロセス管理の不明確さ

### 【中優先度】設計段階で明確化推奨
3. Firewall Automation の実装場所の曖昧さ
4. IPC 実装の詳細不足
5. エラーハンドリングとログ管理の統一性不足
6. 証明書管理の詳細不足

### 【低優先度】実装時に詳細化可能
7. ストリーミング処理の実装詳細不足
8. セキュリティポリシーの実装詳細不足
9. エンジン検出キャッシュの整合性問題（WAL モードで解決可能）
10. テスト戦略の具体性不足

## 推奨される次のステップ

1. **高優先度問題の解決**
   - データベースマイグレーションの競合対策を `DB_SCHEMA.md` に追記
   - プロキシのプロセス管理の詳細を `PROXY_SPEC.md` に追記

2. **中優先度問題の明確化**
   - Firewall Automation の実装場所を明確化
   - IPC 実装の詳細を `UI_MINIMAL.md` に追記
   - エラーハンドリングとログ管理の統一ポリシーを定義

3. **ドキュメントの整合性確認**
   - 各ドキュメント間の参照関係を確認し、矛盾がないか検証
   - 未決事項リストを各ドキュメントに追記

## 結論

計画全体としては、前回の失敗要因（機能肥大化・責務逆転）を踏まえた設計になっており、基本的な方向性は適切です。しかし、実装の詳細レベルでの不明確さが複数存在し、特にデータベースマイグレーションの競合問題とプロキシのプロセス管理については、実装前に明確化が必要です。

これらの問題点を解決することで、計画の破綻リスクを低減し、スムーズな実装が可能になると考えられます。
