# FLM Plan Review - 潜在的問題点の抽出
> Status: Review | Audience: Architecture leads | Updated: 2025-01-XX

## 背景

前回のプロトタイプの失敗要因:
- 機能肥大化による保守困難
- Node/Express と Rust の混在による責務不明確
- UI から CLI subprocess を呼ぶ構造による複雑性

本計画は「Rust コア + 薄いアダプタ」で再構築する方針だが、以下の潜在的問題が存在する。

---

## 1. アーキテクチャ設計上の問題

### 1.1 Domain/Port/Adapter の責務境界が不明確

**問題点**:
- `CORE_API.md` では Domain 層が「純粋ロジック + 抽象ポートのみ」と定義されているが、`EngineService` / `ProxyService` が実際に HTTP/DB 操作をどの程度含むかが曖昧
- `ProxyService` が「Axum サーバ起動を統括し、HTTPS/ACME の証明書ハンドリングも内部に閉じる」とあるが、これは Domain 層の責務を超えている可能性がある

**リスク**:
- Domain 層が実質的に Application 層の責務を担い、前回と同じ「肥大化」を招く
- テスト可能性が低下（HTTP/DB 依存が Domain に混入）

**推奨修正**:
- Domain 層は「状態遷移ロジック」「バリデーション」「ビジネスルール」のみに限定
- HTTP サーバ起動・証明書管理は `flm-proxy` 側の Adapter 層に移動
- `ProxyController` trait をより抽象化し、Domain は「起動要求」のみを扱う

### 1.2 エンジン検出のキャッシュ戦略が複雑

**問題点**:
- `ENGINE_DETECT.md` で「キャッシュ TTL 30秒」とあるが、`EngineService` がキャッシュ管理を担うのか、`ConfigRepository` が担うのかが不明確
- `--fresh` オプションでキャッシュ無視が可能だが、UI からの呼び出し時にキャッシュ制御がどうなるか未定義

**リスク**:
- キャッシュ不整合による状態の不正確な表示
- UI と CLI で異なるキャッシュ戦略が適用される可能性

**推奨修正**:
- キャッシュ管理を `ConfigRepository` の責務として明確化
- `EngineService::detect_engines` は常に `ConfigRepository` 経由でキャッシュを参照し、Domain 層はキャッシュ実装を知らない

### 1.3 SecurityPolicy の JSON スキーマ管理が脆弱

**問題点**:
- `CORE_API.md` では「Phase1/2では SecurityPolicy はグローバルID `"default"` のみ」とあるが、JSON スキーマのバリデーションが Domain 層で行われるのか、UI/Proxy 側で行われるのかが不明確
- `PROXY_SPEC.md` では「Proxy / UI / CLI は `SecurityPolicy.raw_json` に以下のキーが存在する前提で動作する」とあるが、スキーマ変更時の後方互換性が考慮されていない

**リスク**:
- スキーマ変更時に Proxy/UI/CLI が不整合を起こす
- バリデーション漏れによるセキュリティホール

**推奨修正**:
- Domain 層に `SecurityPolicySchema` を定義し、バリデーションロジックを集約
- スキーマバージョン管理を導入し、後方互換性を保証

## 2. 実装の複雑さと現実性の問題

### 2.1 Proxy のストリーミング実装が複雑すぎる

**問題点**:
- `PROXY_SPEC.md` で「Ollama は `ChatStreamChunk` に変換 → Proxy が OpenAI SSE 形式に整形」とあるが、エンジンごとに異なるストリーミング形式を統一する処理が複雑
- `EngineService::chat_stream` が `Pin<Box<dyn Stream<...>>>` を返すが、エラーハンドリングとリソース管理が不明確

**リスク**:
- ストリーミング実装のバグが発生しやすい
- メモリリークやリソースリークの可能性

**推奨修正**:
- Phase1 ではストリーミングを「実験的機能」として位置づけ、非ストリーミングを優先
- ストリーミング実装は Phase2 以降に延期し、まずは同期レスポンスで動作確認

### 2.2 エンジン検出の状態遷移が複雑すぎる

**問題点**:
- `CORE_API.md` の `EngineStatus` 遷移規則が 8 パターン以上あり、実装が複雑
- 「連続3回のヘルスチェック成功」などの閾値管理が `EngineServiceConfig` に依存するが、設定のデフォルト値と変更可能性が不明確

**リスク**:
- 状態遷移のバグが発生しやすい
- デバッグが困難（状態が予測不能）

**推奨修正**:
- Phase1 では状態を `InstalledOnly` / `RunningHealthy` / `Error` の 3 種類に簡素化
- 詳細な状態遷移は Phase2 以降に拡張

### 2.3 ファイアウォール自動化の実装が OS 依存で複雑

**問題点**:
- `SECURITY_FIREWALL_GUIDE.md` で Windows/macOS/Linux それぞれのコマンドを定義しているが、Tauri 側で OS 判定・昇格・実行を実装する必要がある
- `UI_MINIMAL.md` では「Tauri ネイティブプラグイン（Rust）として実装」とあるが、OS 別の実装コストが高い

**リスク**:
- OS 別のバグが発生しやすい
- セキュリティリスク（権限昇格の誤用）

**推奨修正**:
- Phase2 ではファイアウォール自動化を「手動ガイド + スクリプト生成」に限定
- 自動適用は Phase3 以降に延期し、まずは手動実行を前提とする

## 3. データベース設計上の問題

### 3.1 2つの SQLite DB の分離が過剰

**問題点**:
- `config.db` と `security.db` を分離しているが、実際には `proxy_profiles` が `config.db` にあり、`certificates` が `security.db` にあるなど、分離基準が不明確
- 2つの DB を同時に操作する際のトランザクション管理が未定義

**リスク**:
- データ不整合（片方の DB だけ更新される）
- マイグレーションの複雑化

**推奨修正**:
- Phase1 では 1つの SQLite DB (`flm.db`) に統合し、テーブルレベルで分離
- Phase2 以降で必要に応じて分離を検討

### 3.2 マイグレーション実行タイミングが曖昧

**問題点**:
- `DB_SCHEMA.md` で「CLI 起動時」「Proxy 起動時」「UI 起動時」にマイグレーションを実行するとあるが、複数のプロセスが同時に起動した場合の競合が未考慮
- `--no-migrate` フラグが「将来のために予約」とあるが、実際の使用ケースが不明確

**リスク**:
- DB ロックによる起動失敗
- マイグレーション競合によるデータ破損

**推奨修正**:
- マイグレーション実行時に DB ロックを取得し、他のプロセスは待機
- `--no-migrate` の使用ケースを明確化（例: 読み取り専用モード）

---

## 4. テスト要件の厳しさと現実性

### 4.1 Phase1 の合格基準が過剰

**問題点**:
- `PLAN.md` で「エンジン検出成功率100%（対象エンジン4種×主要OSで3回以上）」「ストリーミング負荷テスト（100 req/min）を成功させる」など、Phase1 の合格基準が厳しすぎる
- 4種のエンジン × 3OS × 3回 = 36回の検証が必要だが、CI 環境で全エンジンを用意するのは現実的でない

**リスク**:
- Phase1 が完了しない
- テスト環境構築に時間がかかりすぎる

**推奨修正**:
- Phase1 では Ollama のみを対象とし、他エンジンは Phase2 以降に拡張
- 負荷テストは Phase2 以降に延期

### 4.2 UI テストの自動化が不十分

**問題点**:
- `PLAN.md` で「UI は主要操作3ケースの手動テスト手順を残し、IPC経路をユニットテスト」とあるが、手動テストのみでは回帰が発生しやすい
- IPC 経路のユニットテストだけでは、UI と Core の統合が検証できない

**リスク**:
- UI の回帰バグが発見されにくい
- Phase2 以降の機能追加時に既存機能が壊れる

**推奨修正**:
- Phase2 では IPC 経路の統合テストを追加（Tauri の IPC をモック化）
- E2E テストは Phase3 で導入

---

## 5. セキュリティ実装上の問題

### 5.1 API キーのローテーション戦略が不明確

**問題点**:
- `CORE_API.md` で「`rotate_api_key` は旧キーを即座に `revoked_at` 付きで無効化し、新しいキーID/平文を返す（グレース期間は設けない）」とあるが、ローテーション中のリクエストが失敗する可能性がある
- クライアント側でのキー更新手順が未定義

**リスク**:
- ローテーション時にサービスが一時的に利用不能になる
- クライアント側でエラーが発生し、ユーザー体験が悪化

**推奨修正**:
- Phase1/2 ではローテーション機能を「実験的」として位置づけ、まずは手動でのキー再生成を推奨
- グレース期間の実装は Phase3 以降に検討

### 5.2 IP ホワイトリストの CIDR サポートが複雑

**問題点**:
- `PROXY_SPEC.md` で「CIDR/IPv4/IPv6 文字列の配列」とあるが、CIDR パースとマッチングロジックが Domain 層で実装されるのか、Proxy 側で実装されるのかが不明確
- IPv6 のサポートが必須なのか、オプションなのかが不明確

**リスク**:
- CIDR パースのバグによるセキュリティホール
- IPv6 サポートの実装コストが高い

**推奨修正**:
- Phase1 では IPv4 のみをサポートし、CIDR は `/24` などの簡単な形式に限定
- IPv6 と複雑な CIDR は Phase2 以降に拡張

## 6. 仕様書間の整合性の問題

### 6.1 モデル ID の正規化ルールが不一致

**問題点**:
- `CLI_SPEC.md` で「モデルIDは `flm://{engine_id}/{model_name}` に正規化」とあるが、`PROXY_SPEC.md` では「Phase1/2 は `model` に `flm://engine/model` 形式を必須」とあり、形式が微妙に異なる
- UI 側での表示形式が `UI_MINIMAL.md` で「`flm://engine/model` 形式で表示」とあるが、内部表現との変換が不明確

**リスク**:
- モデル ID の不一致によるエラー
- UI/CLI/Proxy 間でモデル ID が正しく伝播しない

**推奨修正**:
- モデル ID の形式を `flm://{engine_id}/{model_name}` に統一し、全仕様書で明記
- Domain 層に `ModelId` 型を定義し、正規化ロジックを集約

### 6.2 エラーハンドリングの形式が不一致

**問題点**:
- `CLI_SPEC.md` で「共通形式: `{"error":{"code":"ENGINE_NOT_FOUND","message":"..."}}`」とあるが、`CORE_API.md` では `EngineError` enum が定義されており、CLI 出力形式との対応が不明確
- Proxy 側のエラー形式が `PROXY_SPEC.md` で未定義

**リスク**:
- エラー形式の不一致によるクライアント側の処理困難
- デバッグが困難

**推奨修正**:
- Domain 層に `ErrorCode` enum を定義し、CLI/Proxy/UI で共通利用
- エラー形式を `docs/ERROR_SPEC.md` として統一仕様を定義

---

## 7. フェーズ間の依存関係の問題

### 7.1 Phase0 の成果物が Phase1 に依存しすぎている

**問題点**:
- `PLAN.md` で Phase0 に「エンジン検出仕様 (`docs/ENGINE_DETECT.md`)、DBスキーマ＋マイグレーション (`docs/DB_SCHEMA.md`)、プロキシ仕様 (`docs/PROXY_SPEC.md`) を完成させる」とあるが、これらの仕様は Phase1 の実装を通じて初めて検証可能
- Phase0 で仕様を「完成」させても、実装時に矛盾が発見される可能性が高い

**リスク**:
- Phase0 で作成した仕様が実装時に変更され、仕様書と実装の乖離が発生
- Phase0 が長引き、Phase1 が遅延

**推奨修正**:
- Phase0 では「初版仕様の作成」に留め、「完成」は Phase1 の実装完了後に宣言
- Phase1 の実装を通じて仕様を検証し、必要に応じて更新

### 7.2 Phase2 の UI 実装が Phase1 の CLI に依存しすぎている

**問題点**:
- `UI_MINIMAL.md` で「Rust Core API を IPC 経由で呼び出し、CLI subprocess の起動は禁止」とあるが、Phase1 で CLI が完成していない場合、UI の実装が困難
- UI の IPC 実装が Phase1 の CLI 実装と独立しているかが不明確

**リスク**:
- Phase1 の遅延が Phase2 に直結
- UI と CLI で異なる Core API の使い方が発生する可能性

**推奨修正**:
- Phase1 で CLI と Core API の実装を並行して進め、Core API を先に確定
- UI の IPC 実装は Core API の確定後に開始

## 8. その他の懸念事項

### 8.1 ドキュメントの更新頻度が不明確

**問題点**:
- 各仕様書に「Updated: 2025-11-18」とあるが、実装時の変更をどのタイミングで反映するかが不明確
- 仕様書間の相互参照が多いが、一方が更新された際に他方を更新する責任者が不明確

**リスク**:
- 仕様書と実装の乖離
- 開発者が古い仕様を参照して実装する

**推奨修正**:
- 仕様書の更新ルールを `docs/README.md` に明記
- 実装時の変更は即座に仕様書に反映することを義務化

### 8.2 前回の失敗要因が完全に解決されていない

**問題点**:
- 前回は「機能肥大化」が問題だったが、本計画でも Phase1 で「エンジン検出」「モデル一覧」「プロキシ起動」「APIキー管理」「セキュリティポリシー」など、多くの機能を実装しようとしている
- 「薄いアダプタ」という方針は良いが、実装範囲が依然として広い

**リスク**:
- Phase1 が完了せず、再度「未完成のまま機能追加」という状況に陥る
- 保守困難な状態が再発

**推奨修正**:
- Phase1 の範囲をさらに縮小し、「エンジン検出 + プロキシ起動（local-http のみ）」に限定
- その他の機能（APIキー管理、セキュリティポリシーなど）は Phase2 以降に延期

## 推奨される修正アクション

### 優先度: 高

1. **Domain 層の責務を明確化**: HTTP/DB 操作を Domain 層から排除し、純粋なビジネスロジックのみに限定
2. **Phase1 の範囲を縮小**: エンジン検出（Ollama のみ）+ プロキシ起動（local-http のみ）に限定
3. **データベースを1つに統合**: `config.db` と `security.db` を `flm.db` に統合
4. **モデル ID 形式を統一**: `flm://{engine_id}/{model_name}` に統一し、全仕様書で明記

### 優先度: 中

5. **エラーハンドリング形式を統一**: `docs/ERROR_SPEC.md` を作成し、CLI/Proxy/UI で共通利用
6. **ストリーミング実装を Phase2 以降に延期**: Phase1 では同期レスポンスのみ
7. **ファイアウォール自動化を Phase3 以降に延期**: Phase2 では手動ガイド + スクリプト生成のみ

### 優先度: 低

8. **仕様書の更新ルールを明確化**: `docs/README.md` に更新ルールを追記
9. **テスト要件を現実化**: Phase1 の合格基準を緩和し、段階的に厳しくする

---

## 結論

本計画は前回の失敗を踏まえた良い方向性を持っているが、以下の点で改善が必要:

1. **アーキテクチャの責務境界をより明確に**: Domain 層の肥大化を防ぐ
2. **実装範囲を段階的に縮小**: Phase1 を最小限にし、確実に完了させる
3. **仕様書間の整合性を確保**: モデル ID、エラー形式などを統一
4. **テスト要件を現実化**: 過剰な合格基準を緩和

これらの修正により、前回の失敗を回避し、段階的な成功を実現できると考える。
