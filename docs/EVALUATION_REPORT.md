# FLM ドキュメント評価レポート（2回目）
> Status: Evaluation | Audience: Project leads | Updated: 2025-01-XX

## 1. 評価概要

本レポートは、`/workspace/docs/` 配下の2回目の文書作成物について、適切性・完全性・実装可能性を評価したものです。アーカイブ済みプロトタイプ（`archive/prototype/`）との比較も含みます。

### 評価対象ドキュメント
- `PLAN.md` - プロジェクト計画とフェーズ定義
- `CORE_API.md` - Rust コア API 仕様
- `CLI_SPEC.md` - CLI コマンド仕様
- `PROXY_SPEC.md` - HTTP(S) プロキシ仕様
- `DB_SCHEMA.md` - データベーススキーマとマイグレーション戦略
- `ENGINE_DETECT.md` - エンジン検出ロジック仕様
- `FEATURE_SPEC.md` - 機能仕様書
- `UI_MINIMAL.md` - 最小 UI 仕様
- `UI_EXTENSIONS.md` - UI 拡張機能（Phase 3 以降）
- `SECURITY_FIREWALL_GUIDE.md` - ファイアウォール設定ガイド
- `diagram.md` - アーキテクチャ図

## 2. 総合評価

### 2.1 強み

#### ✅ アーキテクチャの明確化
- **Domain/Application/Adapter の責務分離**が明確に定義されている
- `flm-core` が純粋ロジック層として独立し、CLI/UI/Proxy が薄いアダプタとして設計されている
- アーカイブ版の「Node/Express 依存」「CLI subprocess 呼び出し」などの技術負債が解消されている

#### ✅ 仕様の一貫性
- ドキュメント間で用語・データモデル・API 設計が統一されている
  - `EngineStatus` の状態定義（`InstalledOnly`, `RunningHealthy`, `RunningDegraded`, `ErrorNetwork`, `ErrorApi`）が `CORE_API.md` と `ENGINE_DETECT.md` で一致
  - `SecurityPolicy` の JSON スキーマが `CORE_API.md`, `PROXY_SPEC.md`, `CLI_SPEC.md` で一貫
  - モデルID正規化（`flm://{engine_id}/{model_name}`）が全ドキュメントで統一

#### ✅ セキュリティ設計の強化
- `security.db` の暗号化キー管理（OS キーチェーン利用）が明確
- APIキーのハッシュ化（Argon2）とローテーション戦略が定義済み
- IPホワイトリスト・CORS・レート制限が最小構成要件として組み込まれている
- `packaged-ca` モードによる証明書自動インストール戦略が Phase 3 で定義されている

#### ✅ 実装可能性の高さ
- Rust の trait ベース設計が具体的に記載されている（`LlmEngine`, `EngineRepository`, `ProxyController` 等）
- エラーハンドリングが型安全に定義されている（`EngineError`, `ProxyError`, `RepoError`）
- マイグレーション戦略（`sqlx::migrate!()`）と失敗時の読み取り専用モードが明確

#### ✅ フェーズ分けの明確さ
- Phase 0/1/2/3 の境界と合格基準が定量化されている
- Phase 1 のサブフェーズ（1A: エンジン検出、1B: Proxy/セキュリティ）による段階的リリース戦略が合理的

### 2.2 改善が必要な点

#### ⚠️ 未実装部分の明確化不足
- **現時点で実装が存在しない**ことを明示する必要がある
  - `crates/flm-core/`, `crates/flm-cli/`, `crates/flm-proxy/` が未作成であることを `PLAN.md` に明記すべき
  - Phase 0 の「Core API 凍結」が完了していない場合、その状態を明確にする

#### ⚠️ バージョニング戦略の詳細不足
- `CORE_API.md` に「v1.0.0 としてタグ付け・署名」とあるが、**実際にタグが作成されているか**不明
- DTO の `version` フィールド（`{"version":"1.0","data":{...}}`）と Core API バージョンの対応関係が曖昧
  - CLI の JSON 出力は `version: "1.0"` だが、Core API のバージョン（`core-api-v1.0.0` タグ）との関係が不明確

#### ⚠️ テスト要件の具体性不足
- Phase 1 の合格基準は記載されているが、**テストコードの構造**（ユニット/統合/E2E の配置）が未定義
- CI ワークフロー（`ci-cli`, `ci-proxy-load`, `ci-acme-smoke`）の実装状態が不明

#### ⚠️ データ移行戦略の検証不足
- `flm migrate legacy` コマンドの仕様はあるが、**アーカイブ版の実際のデータ構造**との対応表が不足
  - 旧 `config.json` のフィールド → 新 `settings` テーブルのマッピング例が必要
  - 旧 API キーの形式 → 新 `api_keys` テーブルへの変換ロジックの詳細が不明

#### ⚠️ エラーハンドリングの網羅性
- `CORE_API.md` のエラー型定義はあるが、**エラーコードと HTTP ステータスコードの対応表**が `PROXY_SPEC.md` に不足
  - 例: `EngineError::NetworkError` → HTTP 502? 503? 504?
  - `ProxyError::AcmeError` → HTTP 500? 502?

### 2.3 アーカイブ版との比較

#### 改善点
1. **技術スタックの統一**: Node/Express から Rust (Axum/Hyper) への移行により、言語統一とパフォーマンス向上が期待できる
2. **責務の明確化**: アーカイブ版の「UI が CLI subprocess を呼ぶ」構造から「IPC 経由で Core API を呼ぶ」構造へ変更され、依存関係が明確化
3. **セキュリティ強化**: アーカイブ版の「設定ファイルベース」から「DB 分割 + 暗号化キー管理」へ移行

#### 懸念点
1. **機能の縮小**: アーカイブ版にあった機能（OAuth2、リモート同期、モデル共有等）が Phase 1/2 では未実装
   - → `UI_EXTENSIONS.md` に「Post-Core Roadmap」として整理されているが、**優先順位と実装時期が不明確**
2. **移行パスの不明確さ**: アーカイブ版ユーザーが新バージョンへ移行する際の**段階的移行戦略**が不足
   - 例: アーカイブ版を並行稼働させながら新バージョンへ段階的に移行できるか？

## 4. 実装可能性評価

### 4.1 技術的実現可能性: ✅ 高い
- Rust の trait ベース設計は標準的なパターンで実装可能
- `sqlx` によるマイグレーション、Axum による HTTP プロキシは実績のある技術スタック
- OS キーチェーン（DPAPI/Keychain/libsecret）の利用も標準的なアプローチ

### 4.2 リスク要因
1. **ACME 証明書取得の複雑性**: `https-acme` モードの実装は `PROXY_SPEC.md` に詳細が不足
   - DNS-01 / HTTP-01 チャレンジの実装詳細が必要
   - タイムアウト（90秒）内での完了保証が技術的に可能か検証が必要
2. **Firewall Automation の権限昇格**: Windows/macOS/Linux での UAC/sudo プロンプト処理が複雑
   - Tauri のネイティブプラグインとして実装する必要があるが、実装例が不足
3. **エンジン検出の並行処理**: 4エンジン × 複数OS での検出テストが Phase 1A の合格基準だが、CI 環境の構築が複雑
