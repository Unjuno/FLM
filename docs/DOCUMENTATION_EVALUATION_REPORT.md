# FLM ドキュメント評価レポート（2回目）
> Status: Evaluation Report | Audience: Project leads | Date: 2025-01-XX

## 1. 評価概要

本レポートは、`/workspace/docs/` 配下のドキュメント（2回目の作成物）について、適切性・完全性・実装可能性を評価したものです。アーカイブ（`archive/prototype/`）は前回の技術負債として参照のみとし、新ドキュメントの品質を独立して評価します。

## 2. 評価観点

- **構造と組織**: ドキュメントの階層・分類・相互参照の適切性
- **明確性**: 仕様の記述が曖昧でないか、実装者が理解できるか
- **完全性**: 実装に必要な情報が網羅されているか
- **一貫性**: ドキュメント間の矛盾・重複がないか
- **実装可能性**: 具体的な実装手順が示されているか
- **保守性**: 将来の変更に対応できる構造か

## 4. 詳細評価

### 4.1 強み

#### ✅ 明確な責務分離
- Domain 層（`flm-core`）と Adapter 層（CLI/UI/Proxy）の境界が明確
- `CORE_API.md` で trait ベースの抽象化が定義され、実装の自由度を保ちつつ契約を明確化

#### ✅ 実装可能性の高さ
- `CLI_SPEC.md` のコマンド例、`PROXY_SPEC.md` のルーティング表、`DB_SCHEMA.md` のテーブル定義が具体的
- エラーコード表（`CLI_SPEC.md` 4章）により、実装時のエラーハンドリングが明確

#### ✅ セキュリティ考慮
- `security.db` の暗号化キー管理（OS キーチェーン）、APIキーのハッシュ化、監査ログの tamper-resistant 設計が明記
- `SECURITY_FIREWALL_GUIDE.md` で OS 別の手動手順が網羅

#### ✅ 段階的アプローチ
- Phase 0/1/2/3 の明確な分割と、各フェーズの合格基準（メトリクス）が定義
- Phase 1 のサブフェーズ（1A/1B）による柔軟なリリース戦略

#### ✅ 後方互換性の考慮
- `CLI_SPEC.md` の JSON 出力に `version` フィールド、`#[non_exhaustive]` の使用方針
- DTO/IPC 互換性ポリシー（`CORE_API.md` 6章）により、将来の拡張に対応

### 4.2 改善が必要な点

#### ⚠️ Phase 0 の「Core API v1.0.0 フリーズ」手順が抽象的
**問題**: `PLAN.md` 95-96行目で「フリーズ手順」が記載されているが、具体的な運用ルール（ADR テンプレート、承認フロー、バージョン番号の付け方）が不明確。

**推奨**: `docs/ADR_TEMPLATE.md` を作成し、変更提案の形式を標準化。また、`docs/VERSIONING_POLICY.md` で SemVer の適用範囲（Core API / CLI / Proxy）を明文化。

#### ⚠️ エンジン検出のタイムアウト値が明記されていない
**問題**: `ENGINE_DETECT.md` に各エンジンの検出タイムアウト（デフォルト 5 秒）が記載されているが、エンジン別の推奨値や調整方法が不明。

**推奨**: `ENGINE_DETECT.md` に「タイムアウト設定」セクションを追加し、エンジン別の推奨値（Ollama: 3秒、vLLM: 10秒等）と `flm config set engine.detection_timeout_ollama 3000` のような設定方法を記載。

#### ⚠️ IPC の型定義（TypeScript ↔ Rust）が不足
**問題**: `UI_MINIMAL.md` で IPC 経路が定義されているが、TypeScript 側の型定義（`ipc.detect_engines()` の戻り値型）や Rust 側の Tauri コマンド定義が不明確。

**推奨**: `docs/IPC_TYPES.md` を作成し、Tauri IPC の型定義（TypeScript interface / Rust struct）を明記。または `UI_MINIMAL.md` に「IPC 型定義」セクションを追加。

#### ⚠️ Fallback ルールの優先順位が不明確
**問題**: `PROXY_SPEC.md` 8章の Fallback ルール表に「warning を出しつつ無視」とあるが、複数の非対応パラメータが同時に指定された場合の処理順序が不明。

**推奨**: `PROXY_SPEC.md` 8章に「処理順序」セクションを追加し、バリデーション → パラメータ変換 → エンジン呼び出しの順序を明記。

#### ⚠️ データフロー図が不足
**問題**: `diagram.md` は静的構造のみで、リクエストが Proxy → EngineService → Engine へ流れる動的なフローが可視化されていない。

**推奨**: `diagram.md` に「リクエストフロー」セクションを追加し、`/v1/chat/completions` の処理ステップ（認証 → ポリシー → ルーティング → EngineService → レスポンス変換）をシーケンス図で表現。

#### ⚠️ テスト戦略の詳細が不足
**問題**: `PLAN.md` のメトリクス（Phase 1A/1B の合格基準）は定義されているが、具体的なテストケース設計（境界値テスト、エラーケース、統合テストのシナリオ）が不足。

**推奨**: `docs/TEST_STRATEGY.md` を作成し、ユニット/統合/E2E のテスト方針、モック戦略、CI での実行順序を定義。

### 4.3 矛盾・不整合の有無

#### ✅ 一貫性の確認
- `CORE_API.md` の `EngineStatus` と `ENGINE_DETECT.md` の状態判定が一致
- `CLI_SPEC.md` のエラーコードと `CORE_API.md` のエラー型が対応
- `DB_SCHEMA.md` のテーブル定義と `CORE_API.md` のデータモデルが整合

#### ⚠️ 軽微な不整合
- `PROXY_SPEC.md` 6章の「HTTPS は `port + 1`」と `CLI_SPEC.md` 3.3章の「`--port` で指定した値は HTTP 用ポート」が一致しているが、`https-acme` モードで HTTP ポートを無効化するかどうかが不明確。

**推奨**: `PROXY_SPEC.md` 6章に「HTTP/HTTPS ポートの関係」セクションを追加し、`https-acme` モードでは HTTP をリダイレクトするか、HTTP を無効化するかを明記。
